### 推荐的结构

| 变化类型     | 建议位置                 | 负责内容                                                 |
| ------------ | ------------------------ | -------------------------------------------------------- |
| 换 MCU       | `PLATFORM/hc32l07x/`     | 厂商库、时钟、中断、GPIO/UART/ADC/Timer 底层实现         |
| 换板、换引脚 | `BOARD/r612h/`           | 引脚归属、复用功能、UART 实例、ADC 通道、休眠状态        |
| 产品功能增减 | `CONFIG/product_r612h.h` | 蓝牙是否启用、D12 是否启用、LED 数量、输入数量、协议选择 |
| 设备行为     | `DEV/`                   | 去抖、传感器换算、LED 驱动、EEPROM 等                    |
| 业务逻辑     | `APP/`                   | 仪表显示、自检、报警、通信业务                           |

核心原则是：例如“蓝牙 UART 使用 UART0、PA9/PA10、115200”这件事，只在板级配置中出现一次。

可以类似这样定义：

```
typedef struct
{
    boolean_t enabled;
    en_hal_uart_id_t instance;
    stc_board_pin_t tx;
    stc_board_pin_t rx;
    uint32_t baudrate;
} stc_board_uart_cfg_t;

static const stc_board_uart_cfg_t g_astBoardUartCfg[] = {
    [BoardUartBluetooth] = {
        .enabled = FEATURE_BLUETOOTH,
        .instance = HalUart0,
        .tx = BOARD_PIN_PA9_AF1,
        .rx = BOARD_PIN_PA10_AF1,
        .baudrate = 115200u,
    },
};
```

UART 初始化、反初始化、协议任务都遍历这张表，不再分别手写 Bluetooth、D12。增加第三个同类实例，主要就是增加一项配置。

### 输入模块也可以真正数据驱动

ABS 和机油压力其实不需要专用分支，可以统一表达为：

```
typedef struct
{
    en_board_pin_id_t pin;
    boolean_t enabled;
    boolean_t active_level;
    boolean_t initial_active;
    uint16_t assert_delay_ms;
    uint16_t release_delay_ms;
} stc_input_cfg_t;
```

对应关系：

- 普通档位：动作延时 10ms，释放延时 10ms。
- ABS：初始报警，报警立即生效，低电平持续 1000ms 后解除。
- 机油压力：初始不报警，低电平持续 1000ms 后报警，高电平立即解除。

这样以后增加同类输入只是增加配置，不需要继续往循环里塞特殊判断。

### 功能关闭与复数实例

建议同时采用两层控制：

- `FEATURE_xxx`：编译期开关，真正节省 ROM/RAM，适合不同产品型号。
- 配置项中的 `enabled`：决定当前板卡是否初始化某个实例。

不要再把 `XxxIdCount` 等同于“所有实例都必须初始化”。应当遍历配置表，只初始化启用项。

主函数也建议收拢为几个生命周期入口：

```
System_Init();
System_Task1ms();
System_Task();
System_Suspend();
System_Resume();
```

具体启用了哪些模块，由产品配置决定。这样关闭 UART 或增加第二个 LED 驱动时，不必同时搜索 `main.c` 的初始化、任务、休眠和恢复代码。

### 换芯片需要现实一点

换 MCU 不可能真的只修改一个引脚文件，因为启动文件、链接配置、厂商驱动、中断名称、时钟树和 Keil Device 都可能不同。当前工程的芯片型号就明确绑定在 [gpio_output.uvprojx (line 17)](/D:/NACL/A_kd/ArtInChip/project/R612/R612H-MCU/MDK/gpio_output.uvprojx:17)。

合理目标不是“整个工程只改一行”，而是：

- APP 和大部分 DEV 不变；
- 新增一套 `PLATFORM/new_mcu/`；
- 新增或修改一份 `BOARD/xxx/`；
- Keil 中切换对应 Target。

### 推荐实施顺序

1. 先确认并处理 PB11 冲突。
2. 建立唯一的板级引脚清单，并加入重复引脚检查。
3. 把 UART、ADC、捕获、TM3100、IIC 的引脚配置移入板级配置。
4. 给 GPIO 配置增加运行态、休眠态属性，去掉分散的休眠引脚数组。
5. 将输入特殊行为改成参数化状态机。
6. 增加产品功能开关和统一生命周期管理。
7. 最后再尝试移植第二种 MCU，用它验证抽象边界是否正确。

因此，我建议你回原始工程进行这项重构，但应当小步迁移：每次只收拢一种配置，编译并上板验证后再继续。不要把当前最终代码整包复制回去，也不要一开始就设计一个包揽一切的“万能框架”。

# 工程架构与配置使用说明

## 1. 改造目的

本次改造解决的核心问题是：同一个硬件事实不应散落在多个模块中。

以前，EEPROM 的 PB6/PB7 同时写在 GPIO 与 IIC 文件中；ADC、捕获和 TM3100 也分别在各自 BSP 文件中写死物理引脚。换板或复制工程时，需要在多个位置搜索和修改，容易产生漏改或两个功能误占同一引脚的问题。

改造后的原则是：

> 产品配置决定“要哪些功能”，板级配置决定“功能接在哪里”，BSP 负责“怎样操作 MCU”，DEV 负责“设备行为”，APP 只处理产品业务。

## 2. 目录职责

| 目录 | 职责 | 通常在什么时候修改 |
| --- | --- | --- |
| `CONFIG/` | 产品功能开关、产品公共参数 | 做不同产品型号、裁剪功能时 |
| `BOARD/` | 唯一引脚清单、ADC/IIC/捕获/TM3100 实例配置 | 换 PCB、换引脚、增加同类实例时 |
| `BSP/` | GPIO、ADC、定时器、低功耗等 MCU 外设操作 | 换 MCU 或增加新的底层外设能力时 |
| `DEV/` | 输入去抖、ADC 传感器处理、EEPROM、LED 面板等设备行为 | 改设备算法或增加设备类型时 |
| `APP/` | 仪表显示、自检、里程、报警等业务逻辑 | 改产品需求时 |
| `source/` | 主循环、中断入口、统一生命周期 | 改系统调度或功耗流程时 |

当前依赖方向为：

```text
APP -> DEV -> BSP -> MCU driver
              ^
CONFIG -> BOARD

main -> system_runtime -> BSP/DEV
```

`main.c` 是组合入口，可以调用各层；普通 APP 代码不应直接使用物理端口、引脚或 BSP ADC ID。

## 3. 产品配置

文件：`CONFIG/product_config.h`

当前开关包括：

- `PRODUCT_FEATURE_EEPROM`：EEPROM 板级实例和生命周期。
- `PRODUCT_FEATURE_ADC`：ADC 引脚、通道和周期任务。
- `PRODUCT_FEATURE_SPEED_RPM`：车速/转速捕获实例和周期任务。
- `PRODUCT_FEATURE_LED_PANEL`：TM3100 引脚和 LED 面板初始化。
- `PRODUCT_NORMAL_FUNCTION`：正常仪表程序；为 0 时进入原有的维护分支。
- `PRODUCT_DEEP_SLEEP_DELAY_MS`：电门关闭后进入深度休眠的延时。

当前正常仪表业务依赖 ADC 电门检测和 LED 面板，因此配置文件使用 `#error` 阻止关闭这两个核心功能。派生产品如果有其他电门来源或没有显示面板，应先修改 APP，再调整依赖检查。

功能开关会控制板级实例和统一生命周期。若 APP 仍然直接调用某功能，想进一步节省 ROM/RAM，还应使用同一个 `PRODUCT_FEATURE_xxx` 包围对应业务，或从 Keil Target 中排除该模块源文件。

## 4. 板级配置

板级接口在 `BOARD/inc/board_config.h`，当前板的具体数据在 `BOARD/src/board_config.c`。

### 4.1 唯一引脚清单

所有物理 `GpioPortX/GpioPinX` 只应出现在 `g_astBoardPinCfg`。BSP 文件不再写死物理引脚。

每个引脚包含：

| 字段 | 含义 |
| --- | --- |
| `enabled` | 当前产品是否使用该引脚 |
| `port/pin` | 唯一物理端口和引脚 |
| `run_mode` | GPIO、模拟模式，或由 ADC/IIC/定时器等外设模块管理 |
| `sleep_mode` | 休眠时保持、切换模拟模式，或恢复默认输出 |
| `dir` | GPIO 输入/输出方向 |
| `pull_up/pull_down` | 上拉与下拉 |
| `open_drain` | 开漏配置 |
| `init_level` | GPIO 输出初始电平 |
| `af` | 外设复用功能 |

`Board_ConfigValidate()` 在启动时检查所有已启用引脚是否存在重复端口/编号。发现冲突后 `SystemRuntime_Init()` 返回错误，主程序保持安全停机，不继续初始化冲突硬件。

### 4.2 外设实例表

外设表不再重复保存端口和引脚，只引用唯一的 `BoardPinId...`：

- `g_astBoardAdcCfg`：ADC 逻辑信号、引脚 ID、ADC 硬件通道。
- `g_astBoardIicCfg`：IIC 实例、SCL/SDA 引脚 ID、目标频率。
- `g_astBoardCaptureCfg`：捕获实例、输入引脚 ID、TIM3 捕获通道。
- `g_stcBoardTm3100Cfg`：TM3100 数据/时钟/锁存/OE 引脚、PWM 定时器、芯片数量。

因此 EEPROM 的 PB6/PB7 只在引脚清单中出现一次，IIC 表只引用对应逻辑 ID。

## 5. 常见修改方法

### 5.1 只更换一个引脚

例如把 IGN 从 PB11 改为另一个支持 ADC 的引脚：

1. 在 `g_astBoardPinCfg[BoardPinIdAdcIgn]` 修改 `port/pin`。
2. 在 `g_astBoardAdcCfg[BoardAdcIdIgn]` 修改对应的 ADC 硬件通道。
3. 检查芯片手册确认该引脚与 ADC 通道匹配。
4. 编译；上电时还会检查新引脚是否与其他已启用引脚冲突。

物理端口不需要再去 `bsp_adc.c`、休眠数组或 APP 中修改。

### 5.2 修改运行态或休眠态

在相应 `g_astBoardPinCfg` 项中修改：

- 普通输入/输出：使用 `BOARD_PIN_GPIO(...)`。
- 未使用引脚：使用 `BOARD_PIN_ANALOG(...)`。
- ADC、捕获、IIC、TM3100：使用 `BOARD_PIN_PERIPHERAL(...)`。
- 休眠转模拟输入：`BoardPinSleepAnalog`。
- 休眠恢复默认输出：`BoardPinSleepDefaultOutput`。
- 休眠保持不变：`BoardPinSleepKeep`。

运行态与休眠态在同一行附近，不再需要维护独立的休眠引脚数组。

### 5.3 增加普通数字输入

1. 在 `en_board_pin_id_t` 增加逻辑引脚 ID。
2. 在 `g_astBoardPinCfg` 增加唯一物理引脚配置。
3. 在 `en_drv_input_id_t` 增加业务输入 ID。
4. 在 `DEV/src/drv_input.c` 的 `s_astDrvInputCfg` 增加一项。
5. APP 通过 `DRV_Input_IsActive()` 读取业务状态。

输入配置支持以下行为：

- `active_level`：高有效或低有效。
- `initialize_from_input`：启动状态取当前输入，还是采用指定初始状态。
- `initial_active`：指定的初始有效状态。
- `assert_delay_ms`：从无效变为有效的确认时间。
- `release_delay_ms`：从有效变为无效的确认时间。
- `enabled`：当前产品是否启用该输入实例。

例如 ABS 可以设置“初始报警、立即报警、低电平持续 1000ms 后解除”；机油压力可以设置“初始不报警、持续 1000ms 后报警、恢复后立即解除”。这两种行为都只需要配置，不需要在 1ms 状态机里增加专用 `if` 分支。

### 5.4 增加 ADC 信号

1. 增加 `BoardPinId...` 并填写唯一引脚。
2. 增加 `BoardAdcId...` 和 `g_astBoardAdcCfg` 项。
3. 在 `bsp_adc.h` 增加对应兼容逻辑名（若 BSP 外部需要使用）。
4. 在 `en_drv_adc_signal_t` 增加 DEV 语义 ID。
5. 在 `s_aenDrvAdcBspMap` 增加 DEV 到 BSP 的映射。
6. APP 只使用 `DrvAdcSignal...`，不要使用 `BspAdcId...`。

BSP 会为启用的 ADC 项自动建立扫描序号映射，因此关闭某个中间通道后，其余通道不需要手工重排结果索引。

### 5.5 增加软件 IIC 实例

1. 为 SCL/SDA 各增加一个板级引脚 ID。
2. 在 `en_board_iic_bus_id_t` 增加总线 ID。
3. 在 `g_astBoardIicCfg` 增加实例，填写两个引脚 ID和目标频率。
4. 在 `bsp_iic.h` 按需要增加对外逻辑名称。
5. 设备驱动通过 `BSP_IIC_InitBus()` 和 `BSP_IIC_GetHandle()` 获取对应实例。

IIC 句柄数组按 `BoardIicBusIdCount` 自动扩展，不再需要为每个实例复制一套收发函数。

### 5.6 增加捕获输入

1. 增加捕获引脚 ID并设置正确 `af`。
2. 在 `en_board_capture_id_t` 增加实例 ID。
3. 在 `g_astBoardCaptureCfg` 选择 TIM3 捕获通道。
4. 若它代表新的设备含义，在 DEV 的捕获回调中增加该通道到设备 ID 的映射。

### 5.7 修改 TM3100 板级连接

修改 `g_stcBoardTm3100Cfg` 引用的逻辑引脚、OE 定时器、PWM 周期或芯片数量。物理端口仍只在唯一引脚清单中修改。

当前工程支持一个 TM3100 串联面板实例；如果以后需要两块完全独立的面板，应把单个 `g_stcBoardTm3100Cfg` 扩展为数组，并把 BSP 的缓存与状态改为按实例保存。

## 6. ADC 分层接口

APP 现在使用 `en_drv_adc_signal_t`：

```c
raw = DRV_ADC_GetAvg(DrvAdcSignalBrightness);
fuel = DRV_ADC_GetResistanceOhm(DrvAdcSignalFuel);
```

DEV 内部的 `s_aenDrvAdcBspMap` 再把业务信号映射到底层 ADC ID。换板导致 ADC 扫描顺序变化时，APP 不需要改变。

## 7. 系统生命周期

统一入口位于 `source/system_runtime.c`：

| 接口 | 功能 |
| --- | --- |
| `SystemRuntime_Init()` | 校验板级配置，并按明确顺序初始化 BSP/DEV |
| `SystemRuntime_Task1ms()` | 调度已启用模块的 1ms 任务 |
| `SystemRuntime_PrepareDeepSleep()` | 关闭 ADC、应用休眠引脚配置、喂狗 |
| `SystemRuntime_RestoreClockAfterWake()` | 恢复系统时钟并喂狗 |
| `SystemRuntime_WakeupIgnCheck()` | 只恢复 IGN 检测所需 ADC 资源 |
| `SystemRuntime_ResumeAfterIgn()` | IGN 有效后恢复 GPIO、捕获、EEPROM、ADC |
| `SystemRuntime_AbortIgnCheck()` | IGN 无效时重新关闭临时 ADC 资源 |

模块顺序仍然显式写出，便于检查电源、GPIO和外设之间的先后依赖；没有采用自动函数注册表，避免隐藏初始化顺序。

## 8. 换 MCU 的边界

当前 `BOARD` 配置仍使用 HC32L130 厂商枚举，例如 `GpioPortB`、`AdcExInputCH18`，它属于“当前 MCU + 当前 PCB”的板级实现。

换 MCU 时合理的目标是：

1. 保留 APP 和大部分 DEV。
2. 为新 MCU 提供相同语义的 BSP 接口。
3. 为新板建立新的 `board_config.c`。
4. 在 Keil 中建立新 Target，切换启动文件、链接配置、Device Pack、BSP 和 BOARD 源文件。

换 MCU 不可能只改一个宏，因为启动代码、中断名、时钟树和链接地址也会变化；本结构的优势是把这些变化限制在 BOARD/BSP/工程 Target，避免传播到业务层。

## 9. 设计优势

- 物理引脚只有一个来源，换引脚时搜索范围明确。
- 启动时检查重复引脚，能尽早发现类似 PB11 被两个功能占用的问题。
- 运行态与休眠态相邻配置，降低漏改低功耗状态的概率。
- 外设实例引用逻辑引脚，避免 EEPROM 引脚在 GPIO/IIC 中重复描述。
- DEV 使用业务语义 ID，APP 不依赖 ADC 扫描顺序和 BSP 枚举。
- 输入状态机支持不同动作/释放延时，新增同类输入不再复制特殊分支。
- 初始化、周期任务、休眠和恢复集中，关闭模块时检查位置更少。
- 配置表按实例扩展，增加 ADC、IIC、捕获等同类实例时主要增加数据项。
- 初始化顺序保持显式，便于嵌入式调试和功耗问题定位。

## 10. 修改后的检查步骤

每次修改 BOARD 或 CONFIG 后建议依次执行：

1. 检查引脚是否支持目标外设和复用功能。
2. 检查 `enabled` 与产品功能开关是否一致。
3. 使用 Keil 编译，确保 0 Error、0 Warning。
4. 上板检查上电默认电平。
5. 检查所有输入、ADC、通信和显示功能。
6. 实测进入休眠后的引脚状态与功耗。
7. 检查唤醒后的 GPIO、捕获、EEPROM和 ADC 是否恢复。

当前 Keil 工程已加入 `CONFIG`、`BOARD/inc` 包含路径，以及 `board_config.c`、`system_runtime.c` 源文件。

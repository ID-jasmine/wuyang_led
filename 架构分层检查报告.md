# 架构分层检查报告

生成日期：2026-05-29

## 1. 总体结论

当前工程的分层方向是正确的，主体依赖链路基本保持为：

```text
source/main.c
  -> DEV/DRV 逻辑设备层
    -> BSP 板级抽象层
      -> 芯片库/寄存器/具体 IO
```

这符合“后续换芯片、换驱动、换 IO 时尽量少改业务逻辑”的目标。  
目前真正和芯片强绑定的内容主要集中在 `BSP/src`，例如 GPIO 端口、ADC 通道、TIM3 输入捕获、RTC 时钟源、低功耗入口、TM3100 控制脚等。

主要可继续优化的点是：

1. `source/main.c` 仍直接调用了一些 BSP 接口，例如 GPIO 电源控制、低功耗、系统 tick、WDT、TIM3 IRQ 分发。
2. 部分 `DEV/inc` 头文件暴露了 BSP 类型，例如 `drv_adc.h` 暴露 `en_bsp_adc_id_t`，`drv_input.h` 暴露 `bsp_gpio.h`，`led_panel.h` 暴露 `bsp_tm3100_led.h`。
3. 部分 BSP 头文件暴露芯片库类型，例如 `bsp_iic.h` 暴露 `gpio.h` 的端口/引脚类型，`bsp_rtc.h` 暴露 `rtc.h` 的 `stc_rtc_time_t`。

这些不是结构性问题，但如果目标是“换芯片最小改动”，建议下一阶段把对外 API 的类型再收口一层。

## 2. 当前目录分层

```text
BSP/
  inc/    板级接口头文件
  src/    芯片库适配、具体 IO、外设配置

DEV/
  inc/    设备/逻辑驱动接口
  src/    设备抽象、传感器/输入/显示面板逻辑

source/
  main.c  系统初始化、主循环、周期任务、中断入口分发

common/
driver/
  当前扫描到的业务代码主要集中在 BSP、DEV、source
```

## 3. BSP 层职责

### 3.1 GPIO：`BSP/inc/bsp_gpio.h`、`BSP/src/bsp_gpio.c`

功能：

- 用 `en_bsp_gpio_id_t` 把业务相关引脚抽象成逻辑 ID。
- 维护逻辑 ID 到真实端口/引脚/方向/上下拉/默认电平的映射。
- 提供 `Bsp_Gpio_Init()`、`Bsp_Gpio_Read()`、`Bsp_Gpio_Write()`。
- 提供 `Bsp_Gpio_InitSleepPins()`，用于低功耗前将部分输入脚切到模拟模式，并恢复默认输出脚状态。

连接关系：

- 被 `DRV_Input` 用于读取输入脚。
- 被 `source/main.c` 直接用于电源控制和低功耗前引脚处理。

换 IO 时主要修改：

- `BSP/src/bsp_gpio.c` 中的 `s_astBspGpioPinCfg`。

### 3.2 ADC：`BSP/inc/bsp_adc.h`、`BSP/src/bsp_adc.c`

功能：

- 抽象燃油、电源电压、水温等 ADC 逻辑通道。
- 初始化 ADC 对应 GPIO 为模拟输入。
- 配置 ADC 扫描序列，启动/停止转换，读取扫描结果。
- 提供休眠后的 ADC/BGR 唤醒和反初始化接口。

连接关系：

- 被 `DRV_ADC` 调用，`DRV_ADC` 在此基础上做 10 次采样平均。

换 ADC 通道/IO 时主要修改：

- `BSP/src/bsp_adc.c` 中的 `s_astBspAdcChannelCfg`。

### 3.3 IIC：`BSP/inc/bsp_iic.h`、`BSP/src/bsp_iic.c`

功能：

- 软件模拟 IIC。
- 当前有 `BspIicBusIdEeprom` 总线。
- 提供 Start、Stop、Send、WaitAck、ReadByte 等基础时序接口。

连接关系：

- 被 `DRV_EEPROM` 组合成 EEPROM 读写协议。

换 EEPROM IIC 引脚时主要修改：

- `BSP/src/bsp_iic.c` 中的 `s_astBspIicHandle`。

### 3.4 RTC：`BSP/inc/bsp_rtc.h`、`BSP/src/bsp_rtc.c`

功能：

- 使能 XTL 和 RTC 外设。
- 初始化 RTC 时间、周期中断、闹钟中断。
- 提供读写时间接口。

连接关系：

- 被 `DRV_RTC` 封装后提供给 `main.c`。
- `source/main.c` 中 `Rtc_IRQHandler()` 负责清中断并置位 1s 标志。

换芯片 RTC 时主要修改：

- `BSP/src/bsp_rtc.c` 和必要时 `BSP/inc/bsp_rtc.h`。

### 3.5 SYS/WDT：`BSP/inc/bsp_sys.h`、`BSP/src/bsp_sys.c`

功能：

- 初始化系统时钟为 RCH 16MHz。
- 配置 SysTick 1ms 节拍。
- 维护系统毫秒 tick。
- 提供阻塞延时、100us 安全延时。
- 初始化和喂 WDT。

连接关系：

- 被 `main.c` 用于系统节拍、10ms/1s 调度、喂狗。
- 被 `DRV_ADC` 用于短延时。
- 被 `DRV_EEPROM` 用于 EEPROM 写入等待。
- 被 `BSP_RTC` 用于 XTL 启振等待。

换芯片时主要修改：

- 系统时钟、SysTick、WDT、Flash wait cycle 等实现。

### 3.6 TIM3 输入捕获：`BSP/inc/bsp_time_capture.h`、`BSP/src/bsp_time_capture.c`

功能：

- 配置 TIM3 为 1MHz 输入捕获。
- 当前将速度和转速输入映射到 TIM3 捕获通道。
- 支持注册回调，捕获中断内把通道和时间戳上报给上层。

连接关系：

- 被 `DEV_SpeedRpm` 注册捕获回调。
- `source/main.c` 中 `Tim3_IRQHandler()` 调用 `BSP_TimeCapture_IRQHandler()`。

换测速/转速输入脚或定时器时主要修改：

- `BSP/src/bsp_time_capture.c` 中的 `s_atim3_cap_pins` 和 TIM3 相关配置。

### 3.7 TM3100 LED：`BSP/inc/bsp_tm3100_led.h`、`BSP/src/bsp_tm3100_led.c`

功能：

- 管理 15 片 TM3100，每片 16 通道。
- 维护 LED 驱动缓存。
- 通过 SDI/CLK/LE/OE 软件时序刷新到芯片。
- 支持按 chip/channel 或线性 index 设置 LED。

连接关系：

- 被 `LedPanel` 作为底层物理 LED 驱动。

换 LED 驱动芯片或控制脚时主要修改：

- 控制脚宏定义、发送时序、芯片数量、通道数量。

### 3.8 LPM：`BSP/inc/bsp_lpm.h`、`BSP/src/bsp_lpm.c`

功能：

- 进入 DeepSleep。
- 唤醒后恢复 RCH 16MHz 系统时钟。

连接关系：

- 被 `source/main.c` 在电门关闭并计时达到阈值后调用。

换芯片低功耗策略时主要修改：

- `BSP/src/bsp_lpm.c`。

## 4. DEV/DRV 层职责

### 4.1 输入驱动：`DEV/inc/drv_input.h`、`DEV/src/drv_input.c`

功能：

- 将 BSP GPIO 逻辑 ID 二次抽象为输入逻辑 ID。
- 处理有效电平转换。
- 处理按键/信号去抖。
- 提供 `DRV_Input_IsActive()` 和 `DRV_Input_ReadRaw()`。

连接关系：

- 下接 `Bsp_Gpio_Read()`。
- 上接 `main.c` 的电门状态判断和低功耗前二次确认。

换输入 IO 时：

- 如果逻辑输入不变，只改 `BSP/src/bsp_gpio.c`。
- 如果输入逻辑数量或有效电平变化，改 `DEV/src/drv_input.c`。

### 4.2 ADC 驱动：`DEV/inc/drv_adc.h`、`DEV/src/drv_adc.c`

功能：

- 初始化 BSP ADC。
- 每 10ms 采样一次。
- 每 10 次样本计算平均值。
- 提供 ADC ready 状态和平均值读取接口。

连接关系：

- 下接 `BSP_ADC_*`。
- 上接 `main.c` 周期任务，后续可被燃油、水温、电压等业务逻辑读取。

注意：

- 当前 `DRV_ADC_GetAvg()` 参数仍然使用 `en_bsp_adc_id_t`，这会把 BSP ADC ID 暴露到上层。

### 4.3 EEPROM 驱动：`DEV/inc/drv_eeprom.h`、`DEV/src/drv_eeprom.c`

功能：

- 基于 IIC 操作表封装 EEPROM 读写。
- 支持跨页写入。
- 写页后延时等待 EEPROM 内部写周期。

连接关系：

- 下接 `BSP_IIC_*` 和 `BSP_SYS_DelayMs()`。
- 上层可直接用 `DRV_EEPROM_ReadBuffer()`、`DRV_EEPROM_WriteBuffer()`。

注意：

- `drv_eeprom.h` 当前暴露了 `bsp_iic.h`、`IIC_Handle_t` 和 `en_bsp_iic_bus_id_t`，对外接口耦合偏多。

### 4.4 RTC 驱动：`DEV/inc/drv_rtc.h`、`DEV/src/drv_rtc.c`

功能：

- 使用 `RTC_Device + RTC_Ops` 形式封装 RTC。
- 提供初始化、读时间、设时间接口。

连接关系：

- 下接 `BSP_RTC_*`。
- 上接 `main.c` 的 1s 时间读取逻辑。

注意：

- `drv_rtc.h` 当前 include `rtc.h`，使上层仍依赖芯片库的时间结构 `stc_rtc_time_t`。

### 4.5 速度/转速：`DEV/inc/dev_speed_rpm.h`、`DEV/src/dev_speed_rpm.c`

功能：

- 注册 BSP 输入捕获回调。
- 根据捕获时间戳计算速度/转速输入频率。
- 使用 1ms 周期任务处理超时和计算窗口。
- 提供频率值和有效性查询。

连接关系：

- 下接 `BSP_TimeCapture_*`。
- 上接 `main.c` 的 SysTick 1ms 任务。

当前设计亮点：

- TIM3、引脚、捕获寄存器细节留在 BSP。
- DEV 层只关心捕获通道和时间戳，结构比较干净。

### 4.6 LED 面板：`DEV/inc/led_panel.h`、`DEV/src/led_panel.c`、`DEV/src/led_panel_map.c`

功能：

- 将物理 TM3100 chip/channel 映射为逻辑 LED。
- 提供逻辑 LED 设置、清屏、全亮、刷新、输出使能。
- `led_panel_map.c` 集中维护逻辑灯到物理通道的映射。

连接关系：

- 下接 `Bsp_Tm3100Led_*`。
- 上接 `main.c` 初始化，以及后续显示/渲染逻辑。

换 LED 排布时主要修改：

- `DEV/src/led_panel_map.c`。

换 LED 驱动芯片时主要修改：

- `BSP/src/bsp_tm3100_led.c`，以及必要时 `led_panel` 的底层适配。

## 5. `main.c` 当前承担的职责

位置：`source/main.c`

当前职责：

- `sys_init()`：初始化系统 tick、GPIO、输入、RTC、速度/转速、LED、ADC、WDT。
- 主循环：处理电门状态变化、电源输出、自检流程占位、ADC 10ms 任务、低功耗进入/唤醒恢复、RTC 1s 读取、1s 喂狗。
- `SysTick_IRQHandler()`：1ms tick 增量、速度/转速 1ms 任务、输入 1ms 任务、电门状态扫描。
- `Rtc_IRQHandler()`：RTC 周期中断和闹钟中断标志处理。
- `Tim3_IRQHandler()`：转发给 BSP 输入捕获中断处理。

当前连接关系：

```text
main.c
  -> DRV_Input: 电门、输入去抖
  -> DRV_ADC: ADC 周期采样
  -> DRV_RTC: 读取时间
  -> DEV_SpeedRpm: 速度/转速频率任务
  -> LedPanel: LED 初始化和后续显示入口
  -> BSP_GPIO: 电源/LED 电源控制、睡眠引脚配置
  -> BSP_LPM: 进入低功耗、唤醒后恢复时钟
  -> BSP_SYS/WDT: tick、延时基准、喂狗
  -> BSP_TimeCapture: TIM3 中断分发
```

## 6. 依赖检查

### 6.1 做得比较好的地方

- BSP 层没有反向 include DEV 层，底层没有依赖上层。
- 具体端口、引脚、芯片外设调用主要集中在 `BSP/src`。
- 输入、ADC、EEPROM、RTC、速度/转速、LED 面板都已有二次封装，不是业务层直接到处操作寄存器。
- LED 面板的逻辑灯和物理 chip/channel 映射已经独立到 `led_panel_map.c`，后续改灯位很方便。
- 速度/转速使用回调把捕获事件从 BSP 上报到 DEV，边界比较自然。

### 6.2 需要继续收口的耦合点

1. `source/main.c` 直接 include BSP：

```text
source/main.c:1  #include "bsp_gpio.h"
source/main.c:2  #include "bsp_lpm.h"
source/main.c:3  #include "bsp_sys.h"
source/main.c:4  #include "bsp_time_capture.h"
```

这说明应用层仍直接管理部分板级资源。建议后续可拆一个 `app_power` 或 `drv_power`，把电源输出、低功耗进入、唤醒恢复收进去。

2. `main.c` 直接操作 BSP GPIO：

```text
source/main.c:38  Bsp_Gpio_Write(BspGpioIdPower, TRUE)
source/main.c:39  Bsp_Gpio_Write(BspGpioIdLEDPower, TRUE)
source/main.c:49  Bsp_Gpio_Write(BspGpioIdPower, FALSE)
source/main.c:84  Bsp_Gpio_InitSleepPins()
```

建议封装成：

```c
DRV_Power_SetMain(boolean_t on);
DRV_Power_SetLed(boolean_t on);
DRV_Power_PrepareSleep(void);
DRV_Power_EnterDeepSleepIfIgnOff(void);
```

3. DEV 层头文件暴露 BSP 类型：

```text
DEV/inc/drv_adc.h:15      DRV_ADC_GetAvg(en_bsp_adc_id_t id)
DEV/inc/drv_input.h:4     #include "bsp_gpio.h"
DEV/inc/drv_eeprom.h:4    #include "bsp_iic.h"
DEV/inc/led_panel.h:4     #include "bsp_tm3100_led.h"
DEV/inc/led_panel_map.h:4 #include "bsp_tm3100_led.h"
```

建议 DEV 对外定义自己的逻辑 ID，例如：

```c
typedef enum
{
    DrvAdcIdFuel,
    DrvAdcIdPower,
    DrvAdcIdWaterTemp,
    DrvAdcIdCount,
} en_drv_adc_id_t;
```

这样应用层完全不用知道 BSP ADC ID。

4. RTC 对外使用芯片库时间结构：

```text
DEV/inc/drv_rtc.h:4 #include "rtc.h"
BSP/inc/bsp_rtc.h:5 #include "rtc.h"
```

如果换芯片后 `stc_rtc_time_t` 不存在，上层会被牵连。建议定义项目自己的时间结构：

```c
typedef struct
{
    uint8_t year;
    uint8_t month;
    uint8_t day;
    uint8_t hour;
    uint8_t minute;
    uint8_t second;
} drv_rtc_time_t;
```

BSP 内部再做芯片结构转换。

5. BSP 头文件暴露 `gpio.h`：

```text
BSP/inc/bsp_iic.h:5          #include "gpio.h"
BSP/inc/bsp_time_capture.h:5 #include "gpio.h"
```

如果这些结构只在 BSP 内部使用，建议把端口/引脚结构从 `.h` 移到 `.c`，对外只暴露 bus id、callback 和 API。

## 7. 建议的目标依赖图

更理想的方向：

```text
APP/main
  -> APP modules
  -> DEV/DRV public API
  -> BSP public API
  -> chip library
```

应用层建议尽量只看这些：

```text
DRV_Input_IsActive()
DRV_ADC_GetAvg()
DRV_RTC_ReadDateTime()
DEV_SpeedRpm_GetFreqMilliHz()
LedPanel_Set()
APP_Power_RequestSleep()
```

应用层尽量不要直接看到这些：

```text
GpioPortA / GpioPin1
Rtc_* / Tim3_* / Adc_*
M0P_ADC / M0P_BGR
BspGpioIdPower
en_bsp_adc_id_t
stc_rtc_time_t
```

## 8. 后续改造优先级

### P1：把 `main.c` 的电源/低功耗 BSP 调用收口

新建或整理一个电源管理模块：

```text
DEV/inc/drv_power.h
DEV/src/drv_power.c
```

职责：

- 主电源输出控制。
- LED 电源输出控制。
- 低功耗前 GPIO 准备。
- 电门关闭确认。
- 进入 DeepSleep 和唤醒后恢复。

收益：

- `main.c` 不再直接碰 `BspGpioIdPower`、`BspGpioIdLEDPower`、`BSP_LPM_*`。
- 换电源控制 IO 时只改 BSP/DRV，不动应用主循环。

### P2：给 ADC 增加 DEV 层逻辑 ID

把 `DRV_ADC_GetAvg(en_bsp_adc_id_t id)` 改为 `DRV_ADC_GetAvg(en_drv_adc_id_t id)`。  
内部维护 `en_drv_adc_id_t -> en_bsp_adc_id_t` 映射。

收益：

- 上层只知道燃油、电源、水温，不知道 BSP ADC 通道。

### P3：RTC 时间结构去芯片化

定义 `drv_rtc_time_t`，DRV 层对外只暴露项目自己的时间结构。  
BSP 层内部再转换为 `stc_rtc_time_t`。

收益：

- 换芯片 RTC 库时，上层业务不用一起改。

### P4：BSP 头文件减少芯片库 include

尽量让 BSP 头文件只暴露项目自己的类型，芯片库头文件留在 `.c` 内部。

收益：

- 编译依赖更干净。
- 换 SDK 时影响范围更小。

## 9. 结论

你的分层目标已经落地了大半：硬件细节基本集中在 BSP，设备逻辑基本集中在 DEV，`main.c` 更多是在做调度。  
现在不是“架构有没有做对”的问题，而是“接口边界还能不能再干净一点”的问题。

如果按上面的 P1-P4 收口，下一次换芯片时，主要改动范围会更接近：

```text
BSP/src/*
BSP/inc/* 的少量接口
个别 DEV/src 映射表
```

而 `source/main.c` 和业务渲染逻辑可以最大程度保持稳定。
